# mt-goal ゴール条件ドキュメントフォーマット

`mt-create-goal`、`mt-goal` が共通で参照するゴール条件ドキュメントの本文フォーマットです。
ゴール条件ドキュメントは対象プロジェクトの `tmp/mt-goal/docs/[status]/yyyymmdd-[goal-name].md` に保存し、ステータスはディレクトリ名を Source of Truth とします。

## 基本ルール

- ゴール条件ドキュメントは日本語で書く。
- ドキュメント本文はこのファイルを Source of Truth とし、各 `SKILL.md` にテンプレート本文を重複させない。
- `## 🎯 /mt-goal 条件`、`## 🧭 方針` は必須とする。
- `## 💭 背景`、`## 📏 測定方法` は、条件の意図や詳細な測定手順を残す必要がある場合に書く。
- ドキュメントは実行手順の細分化ではなく、条件・測定方法・スコープ・完了境界の明確化を目的に書く。
- セクション間で同じ内容を重複させない。各セクションは責務（背景は why、条件は what、測定方法は how to check in detail、方針は how to achieve）に閉じて書く。

## 本文テンプレート

```markdown
# ゴールタイトル

## 💭 背景

なぜこのゴールが必要か（why）だけを書く。現状、問題、経緯、制約条件、関連リソース、前提条件を書く。
how（進め方や測定方法の判断基準）は `## 🧭 方針` / `## 📏 測定方法` に書き、背景には混入させない。
不要な場合はこのセクションごと省略してよい。

## 🎯 /mt-goal 条件

`/mt-goal <condition>` に渡す一行の条件文字列を書く。
測定可能で証明方法が明確な状態として書く。

- ✅ 良い例: `npm test exits with code 0 and no TypeScript errors are reported`
- ❌ 悪い例: `make tests pass`（曖昧）

## 📏 測定方法

1. 条件文字列に含めきれない、詳細な確認手順を書く
2. Checker（small_model）が判定しやすい形で証拠を残す方法を書く
3. 実行するコマンド、確認するファイル、期待される出力や終了コードを具体的にする

## 🧭 方針

1. `/mt-goal` 実行時に守る進め方、優先する判断基準、避けたい進め方を書く
```

## セクション運用

### 💭 背景

背景はこのゴールが必要になった理由（why）を書きます。
現状、問題、経緯、制約条件、関連リソース、前提条件など、判断材料となる事実を残します。

ユーザーが how（進め方や具体策）を明示してきた場合でも、背景セクションには記載せず `## 🧭 方針` / `## 📏 測定方法` に振り分けます。背景セクションには why だけが残るようにします。
達成したい状態は `## 🎯 /mt-goal 条件`、測定方法は `## 📏 測定方法` に書き、背景には書きません。

### 🎯 /mt-goal 条件

`/mt-goal <condition>` に渡す一行の条件文字列を書きます。
以下の原則に従います。

- **測定可能**: Yes/No で判定できる状態にする
- **証明方法が明確**: どのコマンドやファイルで確認するかを含める
- **制約を含める**: スコープ外や例外条件を必要に応じて含める
- **擦り合わせ内容を反映**: `mt-create-goal` で決めた測定方法や方針の本質部分は、この条件文字列に組み込む

条件文字列は Maker（通常モデル）が作業し、Checker（small_model）が判定します。Checker は作業内容を知らないため、条件テキストに必要な証拠を含めてください。

#### 条件文字列と測定方法・方針の関係

`## 🎯 /mt-goal 条件` に書く一行は、それだけで達成可否を判定できるものにします。
`## 📏 測定方法` や `## 🧭 方針` で擦り合わせた内容のうち、判定に必要な本質部分は条件文字列に盛り込みます。
`## 📏 測定方法` は、条件文字列に含めきれない詳細な手順や、Checker が判定しやすい形での証拠の残し方を補足する場所です。

- ✅ 良い例: `npm test exits with code 0 and no TypeScript errors are reported`
  - この条件だけで、終了コードと TypeScript エラーの有無を確認できる
- ❌ 悪い例: `make tests pass`
  - 何をもって「pass」とするか不明瞭

### 📏 測定方法

`## 🎯 /mt-goal 条件` に含めきれない、詳細な確認手順や証拠の残し方を書きます。
実行するコマンド、確認するファイル、期待される出力や終了コードを具体的にします。
Checker が判定しやすい形で証拠を残す方法を明示します。

このセクションは補足的な位置づけです。判定に必要な本質的な情報は `## 🎯 /mt-goal 条件` に含めてください。

### 🧭 方針

方針はチェックリストではなく、実行時の判断基準として書きます。
作業ステップを細かく列挙する代わりに、判断軸とスコープ境界を示します。
実行中の具体的な手段は、条件達成のために AI が自律的に判断します。

内容が少ない場合は小見出しを省略し、フラットなリストで書いてもかまいません。
ユーザーが how（進め方）を明示してきた場合は、`## 💭 背景` に書かず、このセクションへ落とし込みます。
