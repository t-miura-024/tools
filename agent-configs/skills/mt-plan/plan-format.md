# mt-plan 計画フォーマット (GitHub Issue body 形式)

`mt-create-plan`、`mt-run-plan`、`mt-plan` が共通で参照する GitHub Issue body の本文フォーマットです。
計画は GitHub Issue として保存し、ステータスは GitHub Project の Status custom field を Source of Truth とします。

## 基本ルール

- 計画は日本語で書く。
- 計画本文はこのファイルを Source of Truth とし、各 `SKILL.md` にテンプレート本文を重複させない。
- Issue title に計画タイトルを書く。body には `# 計画タイトル` を含めない（重複を避ける、Q5/Q7 決定）。
- `## 💭 背景`、`## ✅ 完了条件`、`## 📦 アウトプット`、`## 🧭 方針`、`## 🐿️ メモ`、`## 🔍 レビュー`、`## 🐢 履歴` は必須セクション。
- `## 💭 背景` は、計画の意図や制約を残す必要がある場合に書く。
- 計画は実行手順の細分化ではなく、目的・ゴール・スコープ・完了条件の明確化を目的に書く。
- 実行時の進め方は `## 🧭 方針` をもとに AI が判断し、ユーザーは成果物と完了条件の充足を確認する。
- 未解決の `🤔 論点:` は、Done 遷移前に解消、方針へ反映、またはスコープ外化する。
- セクション間で同じ内容を重複させない。各セクションは責務（背景は why、方針は how、完了条件は受け入れ基準、アウトプットは成果物の実体）に閉じて書く。
- 旧フォーマット（`tmp/plan/[status]/yyyymmdd-[plan-name].md`）で作成済みの既存計画は GitHub Issue へ移行しない（Q11 決定）。新規計画は本フォーマットに従う。

## 本文テンプレート

```markdown
## 💭 背景

なぜこの計画が必要か（why）だけを書く。現状、問題、経緯、制約条件、関連リソース、前提条件を書く。
how（進め方の判断基準）は `## 🧭 方針` に書き、背景には混入させない。
不要な場合はこのセクションごと省略してよい。

## ✅ 完了条件

1. ユーザーが成果物を受け入れられる状態を書く
2. スコープ内で満たすべき品質・制約・検証観点を書く
3. 何が確認できれば Done と判断できるかを書く

## 📦 アウトプット

1. 成果物の種類や期待される状態をざっくり書く
2. 必要に応じて主要なファイル・文書・確認結果を書く

## 🧭 方針

### 進め方の原則

1. 実行時に守る進め方、優先する判断基準、避けたい進め方を書く

### AI 判断範囲・Human Gate

1. AI が判断してよい範囲を書く
2. ユーザー確認が必要な境界を書く

小見出しは内容が少ない場合に省略してフラットなリストで書いてもよい。

## 🐿️ メモ

- YYYY-MM-DD
    - 💭 背景: 後で判断材料になる前提や制約を書く。
    - 🤔 論点: 未決事項、要確認事項、候補比較を書く。
    - 🧭 指針: 再開時の方針、運用ルール、合意済みの判断を書く。

## 🔍 レビュー

### 1. 指摘のタイトル

| 項目 | 内容 |
|------|------|
| 優先度 | 🚨 must |
| 観点 | 観点名 |
| 対象 | 対象ファイル・箇所 |
| 日時 | YYYY-MM-DD HH:mm |
| 状況 | 未対応 / 対応済み / 対応不要 |

指摘内容の詳細をここに記述する。

## 🐢 履歴

- YYYY-MM-DD HH:mm [status]
    - 実施内容、変更したファイル、検証結果、中断理由などを書く。
```

## セクション運用

### 💭 背景

背景はこの計画が必要になった理由（why）を書きます。
現状、問題、経緯、制約条件、関連リソース、前提条件など、判断材料となる事実を残します。

ユーザーが how（進め方や具体策）を明示してきた場合でも、背景セクションには記載せず `## 🧭 方針` に振り分けます。背景セクションには why だけが残るようにします。
達成したい状態は `## ✅ 完了条件`、成果物の実体は `## 📦 アウトプット` に書き、背景には書きません。

### ✅ 完了条件

完了条件は、ユーザーが成果物の妥当性をレビューするための受け入れ基準として書きます。
Yes/No で判定可能な状態として書き、成果物の存在自体は `## 📦 アウトプット` に分けます。
「よくする」「整理する」だけで終わらせず、スコープ内で満たすべき状態、品質、制約、検証観点を明確にします。
実行手順の細かさではなく、最終状態が確認できることを重視します。

### 📦 アウトプット

アウトプットには成果物の実体（種類、期待される状態、主要な参照先）を書きます。
その状態が満たせているかの判定基準は `## ✅ 完了条件` に分け、ここでは判定軸を書きません。
計画段階でファイル一覧や状態変更を細かく固定しすぎず、実行中に最適な手段を選べる余白を残します。
完了条件の確認に必要な粒度を超えて詳細化しないようにします。

### 🧭 方針

方針はチェックリストではなく、実行時の判断基準として書きます。
作業ステップを細かく列挙する代わりに、判断軸とスコープ境界を示します。
実行中の具体的な手段は、完了条件を満たすために AI が判断します。

役割は 2 軸に整理します。

- **進め方の原則**: 優先順位、進め方の原則、スコープ境界、避けたい進め方
- **AI 判断範囲・Human Gate**: AI が判断してよい範囲、ユーザー確認が必要な境界

内容が少ない場合は小見出しを省略し、1 つのリストにまとめてもかまいません。
ユーザーが how（進め方）を明示してきた場合は、`## 💭 背景` に書かず、このセクションへ落とし込みます。

### 🐿️ メモ

メモは「本文へ昇格していない暫定情報・未解決事項」の作業域として使います。
3 種類のプレフィックスは、それぞれ本文セクションに対応する暫定情報を示します。

- `💭 背景:` … 本文 `## 💭 背景` に昇格させる前の前提・制約
- `🤔 論点:` … 未決事項、要確認事項、候補比較
- `🧭 指針:` … 本文 `## 🧭 方針` に昇格させる前の合意済み判断・運用ルール

各プレフィックスの内容は、本文セクションへ反映できた時点でメモから削除します。
具体的には次のとおりです。

- `💭 背景:` の内容が `## 💭 背景` に反映されたら、メモ側からは削除する。
- `🧭 指針:` の内容が `## 🧭 方針` に反映されたら、メモ側からは削除する。
- `🤔 論点:` は解決した時点で、決定内容を本文または `## 🐢 履歴` に反映してメモから削除する。

メモに残ったままにせず、確定情報は必ず本文へ寄せて、二重記載を防ぎます。

### 🔍 レビュー

レビューで発生した指摘と対応状況を記録します。
指摘ごとに採番し、優先度（must / should / want）、観点、対象、日時、状況をテーブルで記載します。
指摘の内容はテーブル直下に本文として記述します。
状況は「未対応」「対応済み」「対応不要」のいずれかとします。

#### フォーマット

```markdown
## 🔍 レビュー

### 1. 指摘のタイトル

| 項目 | 内容 |
|------|------|
| 優先度 | 🚨 must |
| 観点 | 観点名 |
| 対象 | 対象ファイル・箇所 |
| 日時 | YYYY-MM-DD HH:mm |
| 状況 | 未対応 / 対応済み / 対応不要 |

指摘内容の詳細をここに記述する。
```

### 🐢 履歴

履歴は計画 Issue の変更事実を時系列で残します。
実行結果、メモ更新、検証結果、状態遷移を簡潔に記録します。

`transition-plan.ts` は状態遷移時に `- YYYY-MM-DD HH:mm [target-status] <source-status> から遷移` のエントリを自動追記します。
手動で追記する場合は、同フォーマットで記述してください。

## Issue title と body の役割分担

- **Issue title**: 計画タイトル（日本語、date prefix なし）
- **Issue body**: 本ファイルに定義する Markdown セクション（`# 計画タイトル` を含めない）

## 保存先とメタデータ

- **保存先**: GitHub Issue (対象プロジェクト repo、または外部 repo の場合は `t-miura-024/note` + `external/<repo-name>` label)
- **必須 label**: `kind/plan`
- **Project**: `~/.config/mt-plan/config.json` の `projectId` で指定
- **Status**: Project の Status custom field (`draft` / `refined` / `in-progress` / `done`)
- **同期**: Status と Issue open/closed は完全同期 (`done` = closed, 他 = open)
