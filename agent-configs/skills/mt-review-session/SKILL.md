# セッション振り返り

セッション中のやり取りを振り返り、プロセスの評価・改善提案・ユーザーへのフィードバックを実施する。

分析・評価は、現在の会話コンテキストを `self-fork` で引き継いだ `mt-review-session-reviewer` サブエージェントに委任する。
トランスクリプトファイルの自動探索や最新ファイル推測は行わない。

`self-fork` によりレビュー対象は「この Skill が実行された現在の会話全体」とする。
過去セッションや別チャットのレビューは、そのセッション上で本 Skill を実行する。

## 🏃 ステップ

### 1. self-fork サブエージェント起動

`mt-review-session-reviewer` サブエージェントを `resume: "self"` で起動する。
これにより、現在の会話コンテキスト全体をレビュー対象として引き継ぐ。
起動時は、少なくとも以下を指定する:

```text
subagent_type: "mt-review-session-reviewer"
resume: "self"
readonly: true
```

**self-fork サブエージェント起動が不可能な環境の場合**:
- トランスクリプトファイル探索にはフォールバックしない
- 本 Skill のレビュー仕様（後掲）を executor 自身で現在の会話コンテキストに対して実行し、分析・評価・提案生成を代行する
- 確証バイアスが入りうることを認識した上で、可能な限り客観的に分析する
- 代行した場合も、以降のステップ（結果表示・アクション選択）は同様に進める

prompt に以下の動的情報を含める：

```text
この会話全体をレビューしてください。

- レビュー対象: self-fork で引き継いだ現在の会話コンテキスト全体
- 主対象: ユーザーが `/mt-review-session` を実行する直前までのセッション
- 対象外: `/mt-review-session` 実行後のレビュー実行過程。ただし、レビュー対象境界の誤りなど、今回の振り返り依頼そのものに関わる問題は必要に応じて補足する
- ワークスペース: <現在のワークスペースの絶対パス>
```

加えて、以下のレビュー仕様を prompt に含める。SubAgent 側は Who 定義に限定し、分析手順・出力フォーマットは本 Skill から渡す。

```text
以下の順でレビューしてください。

0. セッション種別の判定
   - 会話コンテキストから使用 Skill / Rule / SubAgent を抽出する
   - Skill 使用あり / Skill 使用なしを判定し、評価観点と改善提案の出し方を分岐する
   - Skill 使用ありの場合は、対象 Skill の目的・手順・確認・出力・例外処理に照らして実行品質を評価する
   - Skill 使用なしの場合は、セッション内に再利用可能なワークフローがあるかを判定する

1. セッション分析
   - セッションの目的を 1 文で定義する
   - 目的の明確度を「明確 / やや曖昧 / 曖昧・複数混在」で判定する
   - 実行タスク、使用 Skill / Rule / SubAgent、手戻り・非効率ポイント、ユーザーの指示パターンを抽出する

2. 目的の評価
   - 達成度: 目的に対して本質的な解決ができたか、目的外の作業が混入していないかを評価する
   - 効率性: 目的達成までの経路、不要な寄り道、より効率的な手段、手戻り原因を評価する

3. プロセスの評価
   - ツール活用: ツール選択・利用・並列化余地を評価する
   - 作業プロセス: 調査・計画・実装・検証の順序と段階性を評価する
   - Skill 実行: Skill 使用時のみ、目的・手順・確認・出力・例外処理の過不足を評価する

4. 評価スコア
   - セッションサマリ直後に評価スコア表を置く
   - 共通観点として、目的適合・効率性・プロセス・ツール活用・改善提案品質を ★5 段階で採点する
   - Skill 使用時のみ、追加観点として Skill実行品質を採点する
   - 評価は絶対評価ではなく、当該セッションの目的に対する相対評価として扱う
   - 各スコアには 1 文の根拠を添える

5. 採点基準
   - `★★★★★`: 完璧。目的に対してほぼ理想的に進められている
   - `★★★★☆`: 良好。大きな問題はなく、軽微な改善余地がある
   - `★★★☆☆`: 及第点だが注意。目的は満たしているが、見直す価値のある問題がある
   - `★★☆☆☆`: 不十分。成果や進め方に明確な改善余地がある
   - `★☆☆☆☆`: 要改善。目的達成やプロセスに大きな問題がある

6. 改善提案
   - Skill / Rule / SubAgent の枠組みに収まる提案だけを出す
   - Skill 使用ありの場合は、既存 Skill 改善・別 Skill への切り出し・Rule 化・SubAgent 化のどれが適切かを区別する
   - Skill 使用なしの場合は、再利用可能なワークフローがある場合のみ、新規 Skill / Rule / SubAgent の候補を理由付きで提案する
   - 根拠が薄い提案、既存定義と重複する提案、今回のセッションから一般化できない提案、実装対象が曖昧な提案は出さない
   - 各提案に番号、レベル（ユーザー / プロジェクト）、カテゴリ（Skill / Rule / SubAgent）、推奨度、種別、対象、内容、理由を含める
   - 提案する場合は、対象ファイルまたは対象定義、変更箇所、想定効果を含める
   - 提案がない場合は「特になし」と明記する

7. ユーザーへのフィードバック
   - 効果的だった指示パターンを先に伝える
   - 指示の改善ポイント、セッション設計の改善ポイントを提案型で書く

出力は以下の Markdown 構成にしてください。
```

サブエージェントは上記仕様に従って分析・評価・提案生成・フォーマット済み出力を一括で実行し、結果を Markdown で返す。

### 2. 結果表示とアクション選択

#### 1. 結果表示

サブエージェントの出力（フォーマット済み Markdown）をそのままユーザーに表示する。

#### 2. 改善提案の選択

出力内の「💡 改善提案」セクションに提案が 1 件以上ある場合、本文で提案リストを番号付きで提示し、実施する提案（複数可）またはスキップをテキストで選んでもらう。

- 提案ごとに番号、タイトル、推奨度、理由を示す
- 「今回はスキップ」が選ばれた場合はセッション終了

#### 3. 実行モードの選択

提案が選ばれた場合、本文で実行モードの選択肢を番号付きで提示する。

選択肢:
- Agent モード（そのまま実装する）
- Plan モード（実装計画を作成し、確認後に実行する）

Agent モードが選ばれた場合は、選択された提案の実装に着手する。
Plan モードが選ばれた場合は、SwitchMode で Plan モードに切り替えてから実装計画を提示する。

### 3. Cursor 設定レビュー

ステップ 2 で提案を実装した結果、Cursor 設定の定義ファイル（Rule / Skill / SubAgent / Hook）を作成・変更した場合に実行する。該当ファイルがない場合はスキップする。

対象ファイルの判定基準: `_cursor/` または `_cursor_user/` 配下の rules / skills / agents ディレクトリ、もしくは hooks.json に変更があった場合。

変更したファイルに対して `mt-review-cursor-config` Skill を実行し、ガイドライン準拠と品質を確認する。

## ✅ 完了条件

- self-fork された現在の会話コンテキストがレビュー対象になっている
- サブエージェントによるセッション分析結果がユーザーに提示されている
- ユーザーが改善提案に対するアクションを選択している（実施またはスキップ）
- 実施を選択した提案が実装されている（該当する場合）
- Cursor 設定ファイルを変更した場合、レビューが完了している

## ⚠️ 例外時の対応

本 Skill が依存する環境機構が利用できない場合のフォールバック方針:

| 依存機構 | 不在時の対応 |
| -------- | ------------ |
| self-fork SubAgent 起動 | ステップ1 の代行手順に従い、executor が現在の会話コンテキストを直接分析する |
| 対話選択ツール | 提案リストをテキストで表示し、ユーザーのテキスト応答で選択を受け付ける |
| `mt-review-cursor-config` Skill | スキップし、ユーザーに手動レビューを促す |
