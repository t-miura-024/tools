# コードレビュー観点・コメント分類

コードレビューで使用するレビュー観点とコメント分類の定義。
差分レビュー（`mt-review-diff`）および SDD コードレビュー（`mt-sdd-implement`）から参照される。

---

## レビュー観点

### マクロレビュー（アーキテクチャレベル）

| 観点       | 確認内容                  |
| -------- | --------------------- |
| ディレクトリ構成 | 既存の構成ルールに沿っているか       |
| 責務分離     | 関数・クラスの責務が適切に分離されているか |
| 依存関係     | モジュール間の依存が適切か         |
| 設計パターン   | 既存パターンと整合性があるか        |
| 背景目的との整合性 | 変更がレビューで採用した背景（目的・制約・意図）と整合しているか（下記4項目） |

#### 背景目的との整合性（確認項目）

マクロレビューで、ステップ 2 で確定した背景要約（セッション文脈・既存 PR 等）に照らして以下を確認する:

- 変更が述べられた目的を達成しているか
- 目的と矛盾する変更がないか
- 目的外の新たな意図が混入していないか
- 目的に対して過剰設計・過少設計になっていないか

### ミクロレビュー（コードレベル）

| 観点        | 確認内容                 |
| --------- | -------------------- |
| 命名        | 変数・関数・クラス名が意図を表しているか |
| ロジック      | 正確性・効率性・エッジケース処理     |
| エラーハンドリング | 例外処理が適切か             |
| ライブラリ     | 利用ライブラリの選定が適切か       |

### 共通観点

| 観点      | 確認内容                          |
| ------- | ----------------------------- |
| 思想・スタイル | 既存コードの思想やスタイルから大きく外れていないか     |
| 本質的解決   | 目的に対して本質的な解決策となる実装か           |
| YAGNI   | 必要最小限の実装となっているか               |
| 認知負荷    | 過度な最適化によって認知負荷を阻害していないか       |
| 過度なDRY  | コンテキストが異なるモジュールを不適切に共通化していないか |
| 🔄 横展開  | 今回の改修を横展開できそうな箇所はないか          |
| AI アンチパターン | AI 生成コードに特有の失敗傾向がないか（詳細は下記「AI アンチパターン」を参照） |

#### AI アンチパターン

AI が生成したコードに特有の失敗傾向を検出する共通観点。AI 生成差分のレビュー時は、上記の観点に加えて以下を必ず確認する。

| 観点              | 確認内容（AI 特有の失敗傾向）                                          |
| --------------- | -------------------------------------------------------------- |
| 仮定の検証 | 実装が実際の要求と一致するか・既存コードベースの規約に合うか・ドメインルールを正しく理解しているか・現実的なエッジケースを考慮したか |
| もっともらしいが間違っている | 構文は正しいが意味が誤っていないか・存在しない API やメソッドを呼んでいないか（幻覚 API）・学習データ由来の非推奨パターンを使っていないか |
| 配線忘れ | 新しいパラメータ・フィールドが呼び出し元から実際に渡されているか（機構だけ実装され未接続でないか） |
| コピペパターン | 同じ危険なパターン・脆弱性が複数箇所に繰り返されていないか・同一ロジックがファイル間で不整合に実装されていないか |
| 冗長な条件分岐 | if/else で同一関数を引数の有無だけ変えて呼び分けていないか（三項演算子・スプレッドで統一できないか） |
| コールバック + 外部変数キャプチャ | 戻り値で返せる値を、コールバック内から外部変数への代入で組み立てていないか |
| レビュー指摘への不適切な対応 | 指摘の「修正」ではなくテスト・ドキュメント追加で対応したつもりになっていないか・指摘と無関係なファイルを変更していないか |
| コンテキスト適合性 | 命名・エラーハンドリング・ログ出力・テストのスタイルが既存コードベースと一致しているか |
| インテグレーション一貫性 | 同種の API 接続・データ取得・型定義が、既存と異なる方式で実装されていないか |
| スコープクリープ | 要求外の機能・早すぎる抽象化・過剰な設定化・ゴールドプレーティング・要求達成に不要な整理/改名/移動がないか・観測可能な契約（文言・戻り値・ログ・公開 API・型名等）を要求外で変更していないか |
| 早すぎるキャッシュ | 明示要求や計測なしに、キャッシュ層・メモ化・ローカル保存を追加していないか |
| デッドコード | 不要になった旧実装・未使用変数・到達不能コード（論理的に到達しない防御コード含む）・未使用 import が残っていないか |
| フォールバック・デフォルト引数の濫用 | 必須データへのフォールバック（`?? 'unknown'` 等）・全呼び出し元が省略するデフォルト引数・catch での空値返却・多段フォールバックで不確実性を隠していないか |
| 未使用コード | 現在どこからも呼ばれない public 関数や、「対称性のため」「将来の拡張のため」のコードを追加していないか |
| 不要な後方互換コード | 明示要求なしに Legacy 値のマッピング・正規化・`@deprecated` を追加していないか・使用箇所のない互換コードを残していないか |
| 決定トレーサビリティ | 自明でない選択が文書化され、理由が妥当で、代替案が検討され、仮定が明示されているか |

## コメント分類

### 種別

| 種別  | 記号  | 説明                  |
| --- | --- | ------------------- |
| 指摘  | ❗   | 問題があり、修正が必要な箇所      |
| 提案  | 💡  | より良くするための改善案（採用は任意） |

### レベル

| レベル | 記号  | 説明            |
| --- | --- | ------------- |
| マクロ | 🌍  | アーキテクチャ・設計レベル |
| ミクロ | 🌲  | コード詳細レベル      |

### 重大度

| 重大度      | 記号  | 説明                       |
| -------- | --- | ------------------------ |
| Critical | 🚨  | マージ前に必須修正（バグ、セキュリティ問題など） |
| Warning  | ⚠️  | 推奨修正（パフォーマンス問題、可読性低下など）  |
| Info     | ℹ️  | 参考情報・ベタープラクティス           |
