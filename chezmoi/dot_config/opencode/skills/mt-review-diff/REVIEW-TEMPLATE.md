## 記入ルール

### 採用した背景（要約）

- レビュー開始時（`mt-review-diff` ステップ2）で確定した目的・制約・意図を記載する
- セッション文脈に加え、ブランチ差分レビューで既存 PR があった場合は PR の内容も反映している
- 総合評価の判断根拠となるため、省略しない
- 背景が未確定のままレビューを進めない
- PR を参照した場合は PR URL を 1 行で添える

### 推奨度の基準

| 星     | 基準                 |
| ----- | ------------------ |
| ★★★★★ | 必ず対応すべき（品質に重大な影響）  |
| ★★★★☆ | 強く推奨（明確なメリットがある）   |
| ★★★☆☆ | 推奨（メリットがデメリットを上回る） |
| ★★☆☆☆ | 検討の余地あり（状況による）     |
| ★☆☆☆☆ | 参考程度（余裕があれば）       |

### 重大度の基準

| 重大度         | 基準                            |
| ----------- | ----------------------------- |
| 🚨 Critical | バグ、セキュリティ問題、データ損失リスク、本番障害の可能性 |
| ⚠️ Warning  | パフォーマンス問題、可読性低下、保守性低下、テスト不足   |
| ℹ️ Info     | ベタープラクティス、リファクタリング案、スタイル改善    |

### 総合評価の基準

| 評価                 | 基準                        |
| ------------------ | ------------------------- |
| ✅ Approve          | 指摘なし、またはすべて軽微な提案のみ        |
| 🔄 Request Changes | ⚠️ Warning 以上の指摘があり、修正が必要 |
| ❓ Comment          | 設計判断や方針について議論・確認が必要       |
| 🛑 Block           | 🚨 Critical な問題があり、マージ不可  |

### 観点の選択

指摘/提案ごとに最も関連する観点を1つ選択：

**🌍 マクロ観点:**

- 🧭 背景目的との整合性
- 🎨 思想・スタイル
- 🎯 本質的解決
- ✂️ YAGNI
- 🧠 認知負荷
- 🔗 過度なDRY
- 🔄 横展開

**🌲 ミクロ観点:**

- 🏷️ 命名
- ⚙️ ロジック
- 🛡️ エラーハンドリング
- 📦 ライブラリ

### 種別・レベルの選択

**種別:**

- ❗ 指摘: 問題があり、修正が必要な箇所
- 💡 提案: より良くするための改善案（採用は任意）

**レベル:**

- 🌍 マクロ: アーキテクチャ・設計レベルの指摘/提案
- 🌲 ミクロ: コード詳細レベルの指摘/提案

### タイトルの形式

タイトルには推奨度の★を先頭につける：

`### [番号]. [推奨度] [タイトル]`

例:

- `### 1. ★★★★★ アーキテクチャの責務分離が不適切`
- `### 2. ★★☆☆☆ より適切な変数名の提案`

### ファイルパスの記法

- **特定行/範囲への指摘**: `startLine:endLine:filepath` 形式のコードブロックを使用し、該当コードを含める
- **ファイル全体への指摘**: マークダウンリンク形式 `[path/to/file.ts](path/to/file.ts)` を使用

### 確認手順の記載（❗ 指摘のみ）

❗ 指摘コメントには、指摘した事象を読み手が確認・再現できるよう「🔍 確認手順」を記載する。

- 番号付きリストで手順を記載する
- 最後のステップで「期待される結果」と「実際の結果（問題事象）」を明示する
- CLI コマンド、URL、操作手順など具体的に記載する
- 💡 提案コメントには記載不要

### コメントの並び順

コメントは推奨度順（★★★★★ → ★☆☆☆☆）にソートして記載する。

### コメントがない場合

レビューコメントがない場合は以下のように記載：

```markdown
## 🗒️ レビューコメント

特にコメントはありません。現状の実装で問題ありません。
```
