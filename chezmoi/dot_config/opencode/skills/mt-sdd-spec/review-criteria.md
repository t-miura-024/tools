# 仕様レビュー観点

## Completeness（網羅性）

| チェック項目 | 判定基準 |
|-------------|----------|
| 要件カバレッジ | `requirements.md` の各要件に対応する仕様が存在するか |
| 正常系の網羅 | 主要なユースケースがすべて記述されているか |
| 異常系・エッジケース | エラー時の振る舞い、境界値、null/空値の扱いが定義されているか |
| 受け入れ基準の完備 | すべての機能仕様に対してテスト可能な受け入れ基準があるか |
| 前提条件の明示 | 暗黙の前提（認証状態、データの存在など）が明記されているか |
| スコープ外の明示 | 意図的に含めないものが明記されているか |

## Feasibility（実現可能性）

| チェック項目 | 判定基準 |
|-------------|----------|
| 技術的実現性 | 現在の技術スタック・アーキテクチャで実装可能か |
| パフォーマンス要件 | 定量的要件（レスポンス時間、スループットなど）が現実的か |
| 外部依存の妥当性 | 外部サービス・ライブラリの依存が過大でないか、安定しているか |
| 仕様間の無矛盾性 | ある仕様が別の仕様の実装を不可能にしていないか |
| データモデルの実現性 | 定義されたデータモデルが既存 DB スキーマと整合するか |
| マイグレーション可能性 | 既存データのマイグレーションが実行可能か |

## Consistency（一貫性）

| チェック項目 | 判定基準 |
|-------------|----------|
| アーキテクチャ整合 | 既存のレイヤー構成・モジュール分割パターンに沿っているか |
| 命名規則の統一 | 新規エンティティ・API の命名が既存の命名規則と統一されているか |
| 用語の一貫性 | 仕様書内で同じ概念に異なる用語を使っていないか |
| API 設計の統一 | 既存 API のリクエスト/レスポンス形式と統一されているか |
| エラーハンドリング統一 | エラーコード・メッセージ形式が既存と統一されているか |
| 類似機能との整合 | 既存の類似機能と矛盾する振る舞いがないか |

## Risk（リスク）

| チェック項目 | 判定基準 |
|-------------|----------|
| 認証・認可 | 必要な認証・認可チェックが仕様に含まれているか |
| 入力検証 | ユーザー入力のバリデーションが定義されているか |
| データ保護 | 機密データの取り扱い（暗号化、マスキング）が考慮されているか |
| パフォーマンスリスク | N+1 クエリ、大量データ処理などのリスクが識別されているか |
| 可用性リスク | 外部サービス障害時のフォールバックが考慮されているか |
| ロールバック | 問題発生時のロールバック手順・データ復旧が考慮されているか |
