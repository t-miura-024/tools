# レビューフレームワーク

## コメント分類体系

すべてのレビューコメントに以下の属性を付与する。

### 種別

| 種別 | 記号 | 説明 |
|------|------|------|
| 指摘 | ❗ | 問題があり、対応が必要 |
| 提案 | 💡 | より良くするための改善案（採用は任意） |

### 重大度

| 重大度 | 記号 | 説明 | 自動修正ループ |
|--------|------|------|----------------|
| Critical | 🚨 | 仕様の根本的な欠陥。このまま進めると重大な問題が発生する | 対象（Writer/Planner に自動委譲） |
| Warning | ⚠️ | 推奨される修正。対応しなくても致命的ではない | Human Gate で人間が判断 |
| Info | ℹ️ | 参考情報・ベタープラクティス | Human Gate で人間が判断 |

### コメントフォーマット

```markdown
#### [連番] コメントタイトル

- **種別**: ❗ 指摘 / 💡 提案
- **重大度**: 🚨 Critical / ⚠️ Warning / ℹ️ Info
- **該当箇所**: 対象の仕様セクション or タスク番号
- **上流変更要否**: No / Yes — 対象: spec.md / implementation-plan.md（変更内容の概要）
- **内容**: 具体的な指摘/提案内容
- **根拠**: なぜこれが問題か / なぜ改善になるか
- **推奨対応**: 具体的な修正案
```

### 上流変更要否の判定基準

「上流変更要否」は、指摘の修正が**レビュー対象の成果物だけでは完結せず、上流の成果物にも変更が必要**な場合に Yes とする。処理プロトコルの詳細は [upstream-change-protocol.md](upstream-change-protocol.md) を参照。

| レビューフェーズ | 上流変更の対象 |
|-----------------|---------------|
| Phase 2（仕様レビュー） | 不要（対象が spec.md 自体のため） |
| Phase 5（計画レビュー） | spec.md のみ |
| Phase 7（コードレビュー） | spec.md / implementation-plan.md |

## 総合判定基準

レビューレポートの総合判定は以下の基準で決定する:

| 判定 | 条件 |
|------|------|
| **Pass** | Critical 指摘なし、Warning 3件以下 |
| **Conditional Pass** | Critical 指摘なし、Warning 4件以上 |
| **Fail** | Critical 指摘が1件以上存在 |
