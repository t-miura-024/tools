# Review Policy

レビュー時に遵守すべきルールを定義する。

## 5 観点

実行完了後、以下の 5 観点で並列レビューを実施する。

1. **本質性・効率性:** 目的に対して本質的で効率的な解決となっているか。
2. **完了条件の充足:** `## ✅ 完了条件` は完全に満たせているか。
3. **スコープの遵守:** スコープ外の対応はしていないか。
4. **方針との整合:** `## 🧭 方針` から大きく外れた対応はしていないか。
5. **アウトプットの品質:** `## 📦 アウトプット` の品質は問題ないか。

## 指摘の評価レベル

- **🚨 must:** 必ず修正しなければならない重大な問題。
- **⚠️ should:** 必須ではないが修正すべき問題。
- **💡 want:** 任意の改善提案。

## レビューコンテキスト

レビュー時には以下をコンテキストとして参照する。

- 計画 Issue（`## ✅ 完了条件`、`## 📦 アウトプット`、`## 🧭 方針`、`## 💭 背景`）。
- 実行ステップで作成・編集したファイル群。
- `## 🐢 履歴` の実行記録。
- 過去のレビュー指摘（`## 🔍 レビュー` セクション）。既出の指摘を踏まえてレビューする。

## 修正ループ

- must 指摘がゼロになるまで自動で修正ループを回す。
- must がゼロになった後は should / want の対応要否をユーザーに確認する。

## レビュー結果の記録

- 各レビューラウンドの結果を Issue body の `## 🔍 レビュー` セクションに記録する。
- 指摘ごとに採番し、優先度、観点、対象、日時、状況をテーブルで記載する。
- `## 🐢 履歴` にラウンドごとの指摘件数サマリを追記する。

## Done 化の確認

- ユーザーが OK と言ったら、done 化して良いか最終確認する。
- ユーザーが明示的に「Done にする」を選択したときだけ done へ遷移する。
