# Plan Issue Body

以下のフォーマットに従って GitHub Issue body を出力する。
Issue title には計画タイトルを含めるため、body には `# 計画タイトル` を含めない。
各セクションの責務と運用ルールは knowledge/plan-format.md を Source of Truth とする。

```markdown
## 💭 背景

{なぜこの計画が必要か（why）だけを書く。現状、問題、経緯、制約条件、関連リソース、前提条件。不要な場合はこのセクションごと省略してよい。}

## ✅ 完了条件

1. {ユーザーが成果物を受け入れられる状態}
2. {スコープ内で満たすべき品質・制約・検証観点}
3. {何が確認できれば Done と判断できるか}

## 📦 アウトプット

1. {成果物の種類や期待される状態}
2. {必要に応じて主要なファイル・文書・確認結果}

## 🧭 方針

### 進め方の原則

1. {実行時に守る進め方、優先する判断基準、避けたい進め方}

### AI 判断範囲・Human Gate

1. {AI が判断してよい範囲}
2. {ユーザー確認が必要な境界}

## 🐿️ メモ

- {YYYY-MM-DD}
    - 💭 背景: {後で判断材料になる前提や制約}
    - 🤔 論点: {未決事項、要確認事項、候補比較}
    - 🧭 指針: {再開時の方針、運用ルール、合意済みの判断}

## 🔍 レビュー

{レビューで発生した指摘と対応状況を記録する。指摘ごとに採番し、優先度、観点、対象、日時、状況をテーブルで記載する。}

## 🐢 履歴

- {YYYY-MM-DD HH:mm [status]}
    - {実施内容、変更したファイル、検証結果、中断理由など}
```

## 出力ルール

- 日本語で書く。
- Grill Phase で確認した内容（{user_inputs}）をもとに、各セクションに振り分ける。
- `## 💭 背景` には why だけを書き、how は `## 🧭 方針` に振り分ける。
- セクション間で同じ内容を重複させない。
- `## 🐿️ メモ` の `🤔 論点` は、未解決事項があれば残し、解決済みのものは本文へ反映して削除する。
- `## 🔍 レビュー` は空セクションとして出力し、実行時に追記する。
- `## 🐢 履歴` は空セクションとして出力し、状態遷移時に `transition-plan.ts` が自動追記する。
