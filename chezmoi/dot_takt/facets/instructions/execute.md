承認済み計画の方針に基づいて、実行手段を判断し作業を進める。

## 手順

1. 計画 Issue body を `gh issue view <number> --json body` で読み込む。
2. `## ✅ 完了条件`、`## 📦 アウトプット`、`## 🧭 方針`、`## 🐿️ メモ`、`## 🐢 履歴` を確認する。
3. 直近の `## 🐢 履歴` と未解決の `## 🐿️ メモ` の `🤔 論点` から再開位置を判断する。
4. 未解決のメモや判断保留があるなら、着手前に `## 🧭 方針` へ取り込む。
5. 完了条件を満たす実行単位を見極める。
6. 各実行単位ごとに、直接実行モードかガイドモードかを判定する。
   - **直接実行モード:** 対象プロジェクト内のファイル作成・編集、コードや Markdown の変更、ローカルで完結する検証コマンドの実行。
   - **ガイドモード:** 外部サービスでの手動操作、ユーザー本人の判断が必要な作業、ログイン・契約・支払い・連絡など AI が直接できない作業。
   - 判定に迷う場合はガイドモードを選ぶ。
7. 直接実行モードの場合:
   - 必要なファイルを Read ツールで確認する。
   - Edit/Write ツールで変更する。
   - 必要なら検証コマンドや linter を実行する。
8. ガイドモードの場合:
   - 今必要なユーザー操作の目的を説明する。
   - ユーザーが行う操作を示す。
   - 必要なら連絡文面、確認項目、チェックリストを作る。
   - ユーザーの完了報告を待つ（`requires_user_input` で待機）。
9. 実行結果を要約し、output-contracts/execution-report.md に従ってレポートを出力する。
10. GitHub Issue body の `## 🐢 履歴` に 1-3 行のサマリを追記する。
    - `gh issue view <number> --json body` で現在の body を読み、`## 🐢 履歴` セクションに追記して `gh issue edit --body-file` で更新。
11. 実行結果をユーザーに確認する。マークダウン記法は使わず、プレーンテキストとする。選択肢概要行は赤字太字（ANSI `\033[1;31m` 〜 `\033[0m`）で出力する。

     ```
     \033[1;31m1. OK、続けて進む [★★★★★]\033[0m

     実行結果に問題なく、次の作業へ進む

     \033[1;31m2. 修正が必要 [★★★☆☆]\033[0m

     実行結果に問題があり、修正したい

     \033[1;31m3. ここで中断する [★★☆☆☆]\033[0m

     進行を止めて後で再開する
     ```

## 終了判定

- 完了条件を満たしたと判断した場合: `[STEP:0]` タグを出力（review へ遷移）。
- ガイドモードでユーザー作業を待つ場合: `[STEP:1]` タグを出力（requires_user_input で待機、execute 自身へ戻る）。
- 進行不可の場合: `[STEP:2]` タグを出力（ABORT）。

## スコープ遵守

- スコープを超える作業が必要と分かったら、勝手に範囲を広げず、継続・別計画への切り出し・スコープ外化のいずれかをユーザーに確認する。
- 計画外のファイル編集や状態遷移が必要になった場合は、実行を止める。
