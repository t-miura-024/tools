実装のアウトプット品質を評価する。

## 手順

1. 計画 Issue の `## 📦 アウトプット` を読み、期待される成果物の状態を把握する。
2. `{previous_response}` の実行レポートと、実行ステップで作成・編集したファイル群を確認する。
3. 以下を評価する:
   - 成果物が `## 📦 アウトプット` に記載された期待される状態を満たしているか。
   - 既存コードベースの規約（命名、フォーマット、アーキテクチャパターン）に従っているか。
   - 明らかなバグ、エラーハンドリングの欠落、テストの不足はないか。
   - 可読性、保守性、一貫性の問題はないか。
4. 明らかなバグ、エラーハンドリングの欠落、テストの不足を must 指摘とする。
5. 可読性、保守性、一貫性の問題を should 指摘とする。
6. 軽微な改善案を want 指摘とする。
7. 過去のレビュー指摘（`## 🔍 レビュー`）を確認し、既出の指摘を踏まえてレビューする。
8. output-contracts/review-verdict.md に従って結果を出力する。

## 終了判定

- must 指摘が 0 件の場合: `approved` タグを出力。
- must 指摘が 1 件以上の場合: `needs_fix` タグを出力。
