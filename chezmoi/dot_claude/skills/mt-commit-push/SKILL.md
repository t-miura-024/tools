# コミット & プッシュ

現在の git 差分を確認し、変更内容に基づいて適切なプレフィクス付きコミットメッセージを自動生成してコミット（& プッシュ）する。

## 🧠 前提知識

- 関連 Skill: mt-check-git-diff（変更内容の確認に使用）
- コミットメッセージのプレフィクス: `add:` / `fix:` / `update:` / `remove:` / `refactor:` / `chore:`

### モード

この Skill は以下の 2 モードで動作する。呼び出し元が明示的にモードを指定した場合はそれに従う。

- **mt-commit-push**（デフォルト）: コミット + プッシュ
- **commit-only**: コミットのみ（プッシュをスキップ）

### スコープ

コミット対象ファイルの範囲を制御する。呼び出し元が明示的に指定した場合はそれに従う。

- **session**: セッション中にエージェントが編集したファイルのみを対象とする
- **all**: git 差分のある全ファイルを対象とする

**デフォルトの決定ロジック**: スコープ指定がない場合、セッション中にファイル編集（Write / StrReplace 等）を行っていれば **session**、行っていなければ **all** を適用する。

## 🏃 ステップ

0. `git branch --show-current` で現在のブランチを確認する。
    - main または master ブランチの場合は警告を表示し、ユーザーに続行確認を求める。
    - ユーザーが中止を選択した場合は処理を終了する。

1. **mt-check-git-diff Skill** を実行して、現在の変更内容を確認する。
    - 変更がない場合は処理を終了する。

2. **スコープに基づく対象ファイルの決定**:
    - **session スコープ**: セッション中に自身が編集したファイルの一覧を特定し、mt-check-git-diff で検出された変更ファイルとの交差集合を「対象ファイル」とする。交差集合が空の場合は「セッションで変更したファイルに git 差分がありません」と報告して処理を終了する。
    - **all スコープ**: mt-check-git-diff で検出された全ファイルを「対象ファイル」とする。

3. 対象ファイルの変更内容を分析して、以下のルールに基づいて適切なプレフィクスを決定する：
    - `add:` - 新規コードの追加が含まれている場合
    - `fix:` - バグ修正、エラー修正、不具合対応などが含まれている場合
    - `update:` - 既存コードの更新・改善、仕様変更などが含まれている場合
    - `remove:` - 既存コードの削除が含まれている場合
    - `refactor:` - リファクタリング、コード整理、構造改善などが含まれている場合
    - `chore:` - パッケージ更新や自動生成ファイルの更新が含まれている場合

    ※判断に迷ったら fix/update のいずれかを適用すること

4. 対象ファイルの変更内容を要約して、プレフィクス付きのコミットメッセージを生成する。
   日本語でコミットメッセージを生成する。変更内容が明確に伝わるよう記述する。

5. 手順 2 で決定した「対象ファイル」のみを `git add` でステージングする。

6. `git commit -m "[プレフィクス] [コミットメッセージ]"` を実行してコミットする。
    - **Shell ツール実行時は必ず `required_permissions: ["all"]` を付与する**。
      理由: このリポジトリの post-commit git hook は `_cursor/` / `_cursor_user/` の変更を検知して同期するため。
      サンドボックス内実行では、`.cursor/` / `~/.cursor/` / `.git/hooks/` への `rsync --delete` が `Operation not permitted` で失敗する。
    - post-commit hook の sync 出力に `Operation not permitted` などが出た場合は、権限付きで `npm run sync:cursor:project` または `npm run sync:cursor:user` を再実行して同期を完了させる。

7. **プッシュ判定**: モードに応じてプッシュの要否を決定する。
    - **commit-only モード**が指定されている場合 → プッシュをスキップし、手順 8 へ進む。
    - **mt-commit-push モード**が指定されている場合 → プッシュを実行する。
    - **モード指定なし**（単独実行）の場合 → 本文で番号付き選択肢を提示して確認する。
        - 選択肢: 「プッシュする」 / 「コミットのみで完了」
        - 「コミットのみで完了」が選ばれた場合 → プッシュをスキップし、手順 8 へ進む。

    プッシュを実行する場合:
    `git push -u origin HEAD` を実行してリモートリポジトリにプッシュする。
    - プッシュに失敗した場合はエラー内容をユーザーに報告する。
        - 認証エラー: 認証情報の確認を促す
        - リモート未設定: リモートリポジトリの設定を促す
        - その他: エラーメッセージを表示

8. 実行結果をユーザーに報告する。
    - コミットのみの場合: コミットハッシュとメッセージを報告
    - コミット＆プッシュの場合: コミットハッシュ・メッセージとプッシュ先を報告
    - **session スコープの場合**: コミット対象外の未コミット変更ファイルが残っていれば、その旨をユーザーに報告する

## ✅ 完了条件

- 対象ファイルがコミットされている
- モードに応じてプッシュが完了している（mt-commit-push の場合）

## 📦 アウトプット

- コミットハッシュ
- コミットメッセージ
- プッシュ先（プッシュした場合）

## ⚠️ 注意事項

- `git commit` および `git push` は Shell ツール実行時に `required_permissions: ["all"]` を付与する。post-commit git hook が `_cursor/` / `_cursor_user/` の変更検知時に `.cursor/` / `~/.cursor/` / `.git/hooks/` への書き込みを行うため、サンドボックス内では同期が失敗する
- 同期失敗時は権限付きで `npm run sync:cursor:project` / `npm run sync:cursor:user` を再実行して整合性を回復する
