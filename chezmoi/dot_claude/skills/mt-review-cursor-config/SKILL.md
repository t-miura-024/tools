# Cursor 設定レビュー

Rule・Skill・SubAgent・Hook の定義ファイルを棲み分けガイドライン（`cursor-config-guideline.md`）に基づいてレビューし、改善提案を出力する。

## 🧠 前提知識

- 棲み分けガイドライン: `_shared/cursor-config-guideline.md`

## 🏃 ステップ

ユーザーの入力に応じて、以下の手順でレビューする。
レビューの実行（ステップ 2〜4）は `mt-cursor-config-reviewer` SubAgent に委譲し、親エージェントはオーケストレーションに専念する。

### 1. レビュー対象の特定（親エージェント）

**特定ファイル指定の場合:**

ユーザーが指定したファイルをレビュー対象とする。

**ディレクトリ一括スキャンの場合:**

指定ディレクトリ配下の定義ファイルを収集する。対象ディレクトリの指定がない場合はユーザーに確認する。

スキャン対象:

| 種類 | ユーザーレベル | プロジェクトレベル |
| ---- | ---- | ---- |
| Rule | `_cursor_user/rules/*.mdc` | `_cursor/rules/*.mdc` |
| Skill | `_cursor_user/skills/*/SKILL.md` | `_cursor/skills/*/SKILL.md` |
| SubAgent | `_cursor_user/agents/*.md` | `_cursor/agents/*.md` |
| Hook | `_cursor_user/hooks.json` | `_cursor/hooks.json` |

### 2. mt-cursor-config-reviewer SubAgent に委譲（親エージェント）

ステップ 1 で特定したレビュー対象ファイルのパスを `mt-cursor-config-reviewer` SubAgent に渡し、レビュー実行を委譲する。

Task ツールの prompt に以下を含める:
- レビュー対象ファイルのパス一覧
- 本 Skill ファイルのパス（SubAgent がチェックリストと出力フォーマットを参照するため）

SubAgent は「レビュー基準」セクション（定義種別の判定 → チェックリスト実行 → 結果出力）を実行し、構造化されたレビュー結果を返却する。

### 3. 改善アクションの選択（親エージェント）

SubAgent から返却されたレビュー結果をユーザーに提示する。

レビュー結果に 🔴 Must Fix または 🟡 Should Fix が含まれる場合、本文で番号付きのアクション選択肢を提示する。

選択肢:
- 指摘を修正する（対象を選択）
- 今回はスキップ

修正が選ばれた場合は、改善案に基づいてファイルを編集する。

## レビュー基準（SubAgent 参照用）

以下はステップ 2 で委譲された SubAgent が実行するレビューの詳細です。
チェックリストの判断基準は `cursor-config-guideline.md` が SoT であり、本セクションはガイドラインの要約として SubAgent の参照用に提供する。ガイドラインが更新された場合はガイドライン側を優先する。

### 定義種別の判定

各ファイルの配置場所とフォーマットから種別（Rule / Skill / SubAgent / Hook）を判定する。

### チェックリスト

各定義ファイルに対して、以下の観点を順にチェックする。

#### A. 種類の妥当性

判断フローに沿って、定義の種別選択が適切か。

| チェック項目 | 不適合のシグナル |
| ---- | ---- |
| イベント駆動の自動処理か？ → Hook | Rule/Skill にライフサイクルイベント処理が含まれている |
| 全セッション例外なし適用の思想・対話姿勢か？ → Rule | Skill/SubAgent にメインエージェントの Who が含まれている |
| 機械的な検証・修正・権限管理で扱える具体制約か？ → Linter / Formatter / Hook / 権限管理 | Rule に具体的なコマンド条件、フォーマット規約、検証手順が含まれている |
| サブエージェントの Who 定義か？ → SubAgent | Rule/Skill にサブエージェントのペルソナが含まれている |
| 上記いずれでもない → Skill | — |

#### B. Rule 最小化原則

Rule に含まれているのが「全セッションで持つべき思想・対話姿勢」または「対象スコープで常時効く必要がある最小制約・発動条件」のみか。

- 手順・ワークフローが混入していないか（→ Skill へ）
- サブエージェントのペルソナ定義が混入していないか（→ SubAgent へ）
- イベント駆動処理が混入していないか（→ Hook へ）
- 機械的な検証・修正で扱える制約が混入していないか（→ Linter / Formatter / Hook へ）
- コマンド権限や環境依存の実行制御が混入していないか（→ 権限管理 / Hook / 実行ハーネスへ）
- `_cursor_user/settings-source/core-rules.md` の場合、エージェントの魂に相当する思想・考え方・コミュニケーションスタイルだけに絞られているか

#### C. SubAgent の Who 特化

SubAgent が「誰が・いつ・何を・何をしない」に限定されているか。

- 詳細な手順・ワークフローが混入していないか（→ Skill へ委譲）
- 前提知識の詳細が混入していないか（→ Skill へ含める）

#### D. Skill の内容適切性

Skill が「What & How」の実行パッケージとして適切か。

- サブエージェントのペルソナ・行動原則が混入していないか（→ SubAgent へ）
- 全セッション適用の制約が混入していないか（→ Rule へ）

#### E. Hook のイベント駆動限定

Hook がライフサイクルイベントに紐づいた自動処理のみか。

- 対話的なワークフローが含まれていないか（→ Skill へ）

#### F. Command 不在

`.cursor/commands/` / `_cursor/commands/` / `_cursor_user/commands/` にファイルが存在しないか。Command は Skill で代替すべき。

#### G. 配置場所とスコープの妥当性

定義ファイルが正しいディレクトリに配置され、ユーザー設定 / プロジェクト設定の選択が適切か。

**ディレクトリの正確性:**

| 種類 | ユーザーレベル | プロジェクトレベル |
| ---- | ---- | ---- |
| Rule | `_cursor_user/rules/` | `_cursor/rules/` |
| Skill | `_cursor_user/skills/*/SKILL.md` | `_cursor/skills/*/SKILL.md` |
| SubAgent | `_cursor_user/agents/` | `_cursor/agents/` |
| Hook | `_cursor_user/hooks.json` | `_cursor/hooks.json` |

**スコープ選択の妥当性:**

| 判定基準 | → ユーザーレベル | → プロジェクトレベル |
| ---- | ---- | ---- |
| 適用範囲 | 全プロジェクト共通で使う | 特定プロジェクト固有 |
| 内容の性質 | 個人のワークフロー・好み・汎用ツール | プロジェクト固有の規約・構造・ドメイン知識 |
| 例 | コミュニケーションスタイル、git操作、SDD | Obsidian互換性、タスク管理、デイリーノート |

チェック項目:
- ユーザーレベルに配置されているが、特定プロジェクトのドメイン知識・構造に依存していないか
- プロジェクトレベルに配置されているが、他プロジェクトでも汎用的に使える内容ではないか

#### H. テンプレート準拠

各種テンプレートのフォーマットに従っているか。

- Rule: `.mdc` 形式、YAML frontmatter（`description`, `globs`）+ Markdown
- Skill: `SKILL.md`、YAML frontmatter（`name`, `description`）+ Markdown
- SubAgent: `.md`、YAML frontmatter（`name`, `description`）+ Markdown
- Hook: JSON 形式（`hooks.json`）

#### I. 定義間の重複

同じ情報が複数の定義に存在していないか。

- Rule と Skill の両方に同じ手順が書かれていないか
- 複数の Skill 間で重複するワークフローがないか
- SubAgent と Skill の責務境界が曖昧になっていないか

#### J. 定義内容の品質

定義の内容が目的に対して本質的かつ効率的かを確認します。

**本質性:**
- 定義の目的（description / 冒頭の説明）に対して、内容が本質的な解決を提供しているか
- 目的とは無関係な記述・ステップが混入していないか
- 本来対処すべき中核の課題を避けて、周辺的・表面的な対応に留まっていないか

**効率性:**
- トークンコストに見合う効果があるか（冗長な記述、過剰な説明がないか）
- 不要なステップ・確認がないか
- 並列化・簡略化の余地がないか
- SKILL.md の場合、500 行以内に収まっているか
- Rule の場合、LLM に常時読ませる価値がある内容だけに圧縮されているか

**スコープの凝集度:**
- 1 つの定義が 1 つの明確な責務に集中しているか
- 複数の関心事が 1 ファイルに混在していないか（分割すべきではないか）
- 逆に、過度に細分化されていないか（統合すべきではないか）

**明確性:**
- 指示・制約を具体的に記述し、曖昧さが残らないか
- エージェントが判断に迷う記述がないか（条件分岐が網羅されているか）
- 用語が一貫しているか

### レビュー結果の出力フォーマット

以下のフォーマットでレビュー結果を出力する。

```markdown
## 🔍 Cursor 設定レビュー結果

### 対象ファイル一覧

| # | ファイル | 種別 | スコープ |
| - | ---- | ---- | ---- |
| 1 | `path/to/file` | Rule | ユーザー |
| 2 | `path/to/file` | Skill | プロジェクト |

### ファイル別レビュー

#### 1. `path/to/file`（種別 / スコープ）

**✅ 適合項目:**
- A. 種類の妥当性: OK
- ...

**⚠️ 不適合・改善提案:**

##### [観点名] 指摘タイトル
- **深刻度**: 🔴 Must Fix / 🟡 Should Fix / 🟢 Nice to Have
- **現状**: 現在の状態を具体的に記述
- **改善案**: 対象ファイルのどのセクションを、どう変更すべきかを具体的に記述
- **根拠**: ガイドラインのどの原則に基づく指摘か

（不適合がない場合は「不適合なし」）

### サマリ

| 指標 | 値 |
| ---- | ---- |
| レビュー対象数 | N 件 |
| 🔴 Must Fix | N 件 |
| 🟡 Should Fix | N 件 |
| 🟢 Nice to Have | N 件 |
| 不適合なし | N 件 |
```

## ✅ 完了条件

- 対象ファイルすべてに対してチェックリスト A〜J のレビューが完了している
- レビュー結果が出力フォーマットに沿って提示されている
- 🔴 Must Fix / 🟡 Should Fix に対するアクションが選択されている

## ⚠️ 注意事項

- チェックリストは機械的に適用するが、文脈を考慮した判断も行う。形式的な準拠だけでなく、定義の意図を理解した上でレビューする
- 不適合の指摘には必ず具体的な改善案を添える。「ダメ」だけの指摘はしない
- 過度に厳密なレビューは避ける。ガイドラインの精神に沿っていれば、軽微な形式差異は 🟢 Nice to Have に留める
- F（Command 不在）は全体で 1 回チェックすれば十分。ファイルごとに繰り返さない
