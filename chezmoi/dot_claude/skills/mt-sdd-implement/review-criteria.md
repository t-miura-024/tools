# 計画レビュー観点

## Spec Alignment（仕様適合）

| チェック項目 | 判定基準 |
|-------------|----------|
| 機能カバレッジ | `spec.md` の各機能仕様に対応するタスクが存在するか |
| 受け入れ基準カバレッジ | 各受け入れ基準を満たすためのタスクが含まれているか |
| テストケースカバレッジ | Backend タスクのテストケースが `spec.md` の受け入れ基準を網羅しているか。正常系・異常系・エッジケースが含まれているか |
| スコープクリープ | 仕様にない作業がタスクに含まれていないか |
| 優先度整合 | 仕様の優先度とタスクの実行順序が整合しているか |
| データモデル対応 | データモデルの変更に必要なマイグレーションタスクが含まれているか |
| 仕様側の問題検出 | 計画と仕様の乖離が「仕様側の問題」に起因する場合（技術的に実現不可能、受け入れ基準の不備、データモデルの矛盾等）、上流変更要否を Yes として報告しているか |

## Architecture（アーキテクチャ）

| チェック項目 | 判定基準 |
|-------------|----------|
| ファイル構成 | 新規ファイルの配置が既存のディレクトリ構造に沿っているか |
| モジュール分割 | 責務の分離が適切で、既存パターンと整合しているか |
| 技術選定 | ライブラリ・フレームワーク機能の選定が妥当か |
| インターフェース設計 | モジュール間のインターフェースが明確に定義されているか |
| 拡張性 | 将来の拡張を不必要に制限する設計になっていないか |

## Task Structure（タスク構造）

| チェック項目 | 判定基準 |
|-------------|----------|
| タスク粒度 | 各タスクが 1 ステップで完結可能な大きさか |
| レイヤー分類 | 全タスクが Infrastructure / Backend / Frontend のいずれかに分類されているか |
| レイヤー順序 | 実行順序が Infrastructure → Backend → Frontend の順序を遵守しているか |
| TDD 適用 | Backend タスクすべてにテストケースが定義されているか |
| 依存関係の正確性 | タスク間の依存が正しく定義されているか |
| 実行順序の妥当性 | 依存関係に基づく実行順序が論理的に正しいか |
| 規模見積もり | S/M/L の推定が妥当か（過大/過小評価がないか） |
| デッドロック | 循環依存が存在しないか |

## Risk/Impact（リスク・影響範囲）

| チェック項目 | 判定基準 |
|-------------|----------|
| 影響範囲の把握 | 変更の影響範囲が正しく特定されているか |
| 破壊的変更 | 後方互換性を壊す変更が明示されているか |
| 高リスクタスク | リスクの高いタスクに対策・注意事項が記載されているか |
| テスト戦略 | 影響範囲をカバーするテスト方針が含まれているか |
| ロールバック戦略 | 問題発生時の切り戻し方法が考慮されているか |

## AI アンチパターン（計画段階）

AI が生成した実装計画に特有の失敗傾向を検出する観点。計画段階で判定できる設計レベルのアンチパターンを確認する（コード固有の検出はコードレビューで実施する）。

| チェック項目 | 判定基準 |
|-------------|----------|
| 仮定の検証 | 計画が実際の要求と一致しているか・既存コードベースの規約やドメインルールを正しく前提にしているか・現実的なエッジケースを考慮しているか |
| スコープクリープ（過剰提供） | 要求外の機能・早すぎる抽象化（単一実装のためのインターフェース化）・過剰な設定化・ゴールドプレーティングがタスクに含まれていないか（Spec Alignment の「スコープクリープ」が仕様逸脱を見るのに対し、ここでは AI の過剰提供傾向を見る） |
| 過剰/過少エンジニアリング | タスクに不要な抽象化レイヤーがないか・現実的なシナリオのエラーハンドリングが欠けていないか |
| 早すぎるキャッシュ | 明示要求や計測根拠なしに、キャッシュ層・メモ化の導入タスクが含まれていないか |
| 不要な後方互換対応 | 明示要求がないのに Legacy 対応・互換レイヤーのタスクが含まれていないか |
| 決定トレーサビリティ | 自明でない技術選定・設計判断が文書化され、理由が妥当で、代替案が検討され、仮定が明示されているか |
