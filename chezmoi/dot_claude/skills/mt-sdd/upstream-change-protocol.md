# 上流変更プロトコル（Upstream Change Request: UCR）

下流フェーズで上流成果物の変更が必要と判断された場合の処理プロトコル。

## UCR の定義

**Upstream Change Request (UCR)** とは、下流フェーズの作業中に発見された、上流成果物の変更が必要な事項を指す。

### 変更対象の分類

| 分類 | 対象成果物 | 例 |
|------|-----------|-----|
| **仕様変更（Spec）** | spec.md | 機能仕様の矛盾、受け入れ基準の不備、データモデルの変更、API 仕様の修正 |
| **計画変更（Plan）** | implementation-plan.md | タスク分割の見直し、依存関係の修正、ファイルパスの変更、テストケースの追加・修正 |
| **両方** | spec.md + implementation-plan.md | 仕様変更に伴い計画も更新が必要なケース |

### 変更伝播マトリクス

| 検出フェーズ | spec.md | implementation-plan.md |
|---|---|---|
| Phase 5（計画レビュー） | Yes | - （対象の成果物自体） |
| Phase 6（実装） | Yes | Yes |
| Phase 7（コードレビュー） | Yes | Yes |
| Phase 8（仕様適合検証） | Yes | Yes |

## 検出トリガー

UCR は以下の 2 つの経路で検出される:

1. **レビューコメント経由**: レビュアーが「上流変更要否: Yes」属性をコメントに付与（[review-framework.md](review-framework.md) 参照）
2. **SubAgent からの明示的フラグ**: Implementation Planner・Implementer 等が作業中に上流成果物の問題を検出し、出力に UCR フラグを含める

## 処理フロー

### Step 1: UCR 集約

オーケストレーターが以下のソースから UCR を抽出・集約する:

- レビューコメントの「上流変更要否: Yes」
- SubAgent の出力に含まれる UCR フラグ（`[UCR]` プレフィックス付きの報告）

各 UCR について以下を整理する:

- 変更対象（spec.md / implementation-plan.md / 両方）
- 変更内容の概要
- 変更の理由・根拠（検出元のコメントやフラグを引用）

### Step 2: ユーザー承認

本文で UCR の一覧と番号付き選択肢を提示し、ユーザーのテキスト回答で判断を求める。

| 選択肢 | 動作 |
|--------|------|
| **全承認** | すべての UCR を承認し、上流成果物を更新する |
| **個別選択** | UCR ごとに承認/却下を選択する |
| **却下** | すべての UCR を却下し、現フェーズを続行する |

### Step 3: 上流成果物の更新

承認された UCR に基づき、対象成果物を担当する SubAgent type に再委譲して更新する:

| 対象 | 担当 SubAgent type | 入力 |
|------|-------------|---------------|
| spec.md | `mt-sdd-spec-writer` | 現在の `spec.md`、承認済み UCR、関連成果物 |
| implementation-plan.md | `mt-sdd-implementation-planner` | 現在の `implementation-plan.md`、承認済み UCR、関連仕様 |

更新指示には以下を含める:

- 承認された UCR の内容（変更箇所・変更内容・理由）
- 変更に伴い整合性を確認すべき関連セクション
- 更新後に親エージェントが再確認すべき成果物

### Step 4: 連鎖更新の評価

上流成果物の更新後、さらに下流の成果物への影響を評価する。

**連鎖更新ルール**:

| 更新した成果物 | 評価対象 | 判断基準 |
|---------------|---------|---------|
| spec.md | implementation-plan.md | 変更が機能仕様・受け入れ基準・データモデル・API 仕様に及ぶ場合、計画のタスク定義・テストケースに影響する可能性がある |
| implementation-plan.md | 実行中のタスク | 変更がタスク定義・依存関係・実行順序に及ぶ場合、残りのタスク実行に影響する可能性がある |

連鎖更新が必要と判断した場合、Step 3 と同様に該当 SubAgent type へ再委譲して更新する。

### Step 5: 変更ログの記録

`appendix-change-log.md` に UCR の処理結果を記録する。

## 変更ログのフォーマット

`appendix-change-log.md` は以下のフォーマットで記録する:

```markdown
# 変更ログ

## UCR-1: <!-- 変更タイトル -->

- **検出フェーズ**: Phase <!-- N -->（<!-- フェーズ名 -->）
- **検出元**: <!-- レビュアー名 or SubAgent 名 -->
- **変更対象**: spec.md / implementation-plan.md / 両方
- **変更内容**: <!-- 具体的な変更内容 -->
- **変更理由**: <!-- なぜこの変更が必要か -->
- **連鎖更新**: あり（<!-- 対象成果物 -->）/ なし
- **ステータス**: 承認 / 却下
```

## オーケストレーターの行動指針

- UCR 処理は Human Gate の**前**に実行する（Human Gate には更新済みの成果物を提示する）
- Critical レビューコメントの自動修正ループと UCR 処理が同時に発生した場合、**UCR 処理を先に実行**する（上流を正した上で自動修正する）
- UCR 処理後にレビューの再実行が必要かはオーケストレーターが判断する（変更の規模による）
    - 軽微な変更（typo 修正、表現の明確化等）: 再レビュー不要
    - 実質的な変更（機能仕様の追加・変更、タスク構成の変更等）: 再レビュー推奨
