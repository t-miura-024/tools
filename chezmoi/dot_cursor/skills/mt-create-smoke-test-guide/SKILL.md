# スモークテスト手順書の作成

## 🧠 前提知識

- 差分取得 Skill: `mt-check-git-diff`（未コミット差分）、`mt-check-branch-diff`（ブランチ差分）
- テスト手順書テンプレート: [test-template.md](test-template.md)

## 🏃 ステップ

以下のステップで構成（ステップ 2・3 は並列実行可能）:

### 1. ヒアリング

本文で番号付き選択肢として以下を確認する:

**差分スコープ**（選択肢）:

- a) 未コミット差分のみ
- b) ブランチ差分のみ
- c) 未コミット差分 + ブランチ差分

**要件ソース**:

- Notion URL があれば提供してもらう
- ない場合は口頭で要件を説明してもらう
- 要件なしでも差分から推論可能（網羅性は下がる）

### 2. 差分の取得 / 3. 要件の取得（並列実行可能）

ステップ 2 と 3 は互いに依存しないため並列実行可能。

**差分の取得**（ステップ 1 で選択されたスコープに応じて）:

- 未コミット差分 → `mt-check-git-diff` Skill を実行
- ブランチ差分 → `mt-check-branch-diff` Skill を実行
- 両方 → 両 Skill を実行

**要件の取得**:

ユーザーが提供した要件ソースから要件を取得する:

- **Notion URL**: Notion MCP（`notion-fetch`）で取得。子ページやサブアイテムも含めて辿る
- **口頭説明**: ユーザーから聞き取り
- **要件なし**: 差分のみから推論する（テストの網羅性は下がる旨をユーザーに伝える）

### 4. 実装内容の分析

SubAgent（explore）で並列に分析する。以下の観点で変更内容を把握する:

**DB/スキーマ変更**:

- 新規テーブル・カラム、制約（一意制約、FK 等）、インデックス

**バックエンド API 変更**:

- 新規/変更エンドポイント、入力バリデーション（Zod 等）、ビジネスロジック（権限チェック、重複チェック、上限チェック等）、エラーレスポンス

**フロントエンド UI 変更**:

- 新規/変更コンポーネント、画面遷移、フォームバリデーション、状態管理

**テストコード**:

- テストケースから期待される振る舞いを抽出

### 5. テスト手順書の生成

要件・差分から機能名を決定し、[test-template.md](test-template.md) のフォーマットに従って `tmp/` 配下に Markdown ファイルを出力する。

ファイル名: `tmp/smoke-test-{機能名のkebab-case}.md`

#### ユーザーストーリー設計の基準

ストーリーはテストケースの網羅性を左右するため、以下の基準を守る。

フォーマット: `{who}として、{what}ができる。{why}のため。`

##### 粒度: 1ストーリー = 1つの独立したユーザーゴール

「この操作だけで完結してユーザーに価値が生まれるか？」を判断基準にする。複数の独立した操作（例: 作成・編集・削除）を 1 ストーリーにまとめない。

- Bad: 「カスタム変数を作成・一覧・編集・削除できる」（4 ゴール混在）
- Good: 「定型文をカスタム変数として登録できる」（1 ゴール）

##### {why}: ユーザーの業務上の恩恵を記述する

技術的な動作（展開される、チェックされる等）ではなく、ユーザーが得る具体的な恩恵を書く。

- Bad: 「カスタム変数がメッセージに展開される」（何が嬉しいか不明）
- Good: 「メッセージごとに同じ情報を手入力する手間を省ける」（業務効率が伝わる）

##### {what}: 手段ではなく成果を記述する

UI 操作やシステム内部動作ではなく、ユーザーが達成したいことを書く。

- Bad: 「タブ切り替えと検索で変数を見つけられる」（UI 仕様の記述）
- Good: 「メッセージ作成時に目的の変数をすぐ見つけて挿入できる」（ゴールの記述）

##### 防御的機能: 安心して操作できる環境として表現する

バリデーション・エラーチェック等は「〜から守られる」ではなく「〜を安心して行える」と表現する。

- Bad: 「重複キーが作成されない」（システム制約の記述）
- Good: 「既存の変数を壊す心配なく新しい変数を追加できる」（安心感が伝わる）

#### テストケース設計の観点

**正常系**:

- CRUD 操作（作成・参照・更新・削除）の各フロー
- 主要なユースケースの End-to-End フロー
- データの表示・反映（一覧表示、並び順、検索・フィルタ）
- UI 操作（コピー、モーダル開閉、タブ切替等）

**準正常系**:

- フロントエンドバリデーション（必須項目、文字数制限、入力形式）
- サーバーサイドバリデーション（重複チェック、上限チェック、権限チェック、予約語チェック）
- エラーメッセージの表示
- disabled 状態の確認
- 関連機能への影響（削除後の参照、値更新後の反映等）

## ✅ 完了条件

- テスト手順書が `tmp/` 配下に出力されている
- 全テストケースに「前提条件」「手順」「期待結果」が記載されている
- ユーザーストーリーと各テストケースの対応表が含まれている
- 正常系・準正常系の両方がカバーされている

## 📦 アウトプット

`tmp/smoke-test-{機能名のkebab-case}.md` — Markdown 形式のスモークテスト手順書

## ⚠️ 注意事項

- 手順は画面上の具体的な操作で記述する（API を直接叩く手順は避ける。ただし UI から操作できないケースは例外）
- 期待結果は手順の番号に対応させて、どのステップで何を確認するか明確にする
- 要件がない場合でも差分から推論してテストケースを作成するが、網羅性が下がる旨をユーザーに伝える
