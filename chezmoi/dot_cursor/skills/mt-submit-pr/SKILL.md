# PR 作成

現在のブランチからベースブランチへのプルリクエストを作成する。

## 🧠 前提知識

- 関連 Skill: mt-check-branch-diff（ベースブランチの検出と差分取得に使用）
- `gh` CLI を使用して PR を作成・更新する
- 説明欄は一時ファイル経由（`--body-file`）で渡す（シェルエスケープ問題の回避）
- Notion MCP（`notion-fetch`）が利用可能な場合は背景情報の自動取得に使用

## 🏃 ステップ

`TodoWrite` ツールを使って以下の TODO リストを作成し、進捗を管理する。

1. `git branch --show-current` を実行して、現在のブランチが feature ブランチもしくは hotfix ブランチかどうかを確認する。
   feature ブランチの条件はブランチ名が「feature/」から始まるブランチかどうかとする。
   hotfix ブランチの条件はブランチ名が「hotfix/」から始まるブランチかどうかとする。
   上記いずれにも該当しない場合はユーザーに伝えて作業を終了する。
2. **mt-check-branch-diff Skill** を実行して、ベースブランチの検出と差分を取得する。
   取得した情報（ベースブランチ名、差分ファイル一覧、変更統計、各ファイルの詳細 diff）を以降の手順で使用する。
3. `git fetch origin [ベースブランチ]` を実行して、ベースブランチを最新の状態に更新する。
4. `git merge origin/[ベースブランチ]` を実行して、feature ブランチにベースブランチを取り込む。
   コンフリクトした場合は解消せず、ユーザーにコンフリクトしてしまったため解消してほしい旨を伝えて作業を終了する。
5. `git push origin [現在のブランチ]` を実行して、変更をリモートにプッシュする。
   プッシュに失敗した場合はエラー内容をユーザーに伝えて作業を終了する。
6. 手順 2 で取得した変更統計をもとに、変更の規模を判定する。
   規模判定は以下の基準を参考にしつつ、差分の内容（DB 変更の有無、アーキテクチャ変更の有無など）も考慮して総合的に判断する。

    | 規模   | 判定基準                                                 |
    | ------ | -------------------------------------------------------- |
    | 小規模 | 行数が少なく、変更が単純（1〜2ファイル程度の軽微な修正） |
    | 中規模 | ある程度の行数があり、複数ファイルにまたがる変更         |
    | 大規模 | 大量の行数、または複雑なアーキテクチャ変更を含む         |

7. `gh pr list --head [現在のブランチ名]` を実行して、プルリクエストがすでに存在するかどうかを確認する。
8. ユーザーにこの PR の背景情報を確認する。
    - **Notionなどのドキュメントがある場合**:
        1. Notion MCP（`notion-fetch` ツール）が利用可能か確認する
        2. 利用可能な場合 → ユーザーに URL を聞き、`notion-fetch` で内容を自動取得する。取得失敗時はペーストを依頼する
        3. 利用不可の場合 → ユーザーに **リンクと内容のペーストを同時に** 1 回で依頼する（例: 「Notion のリンクと内容をまとめてペーストしてください」）
        4. 取得した内容を簡潔に要約して背景セクションに記載する（詳細はリンクから確認できるため要約で十分）
    - **ドキュメントがない場合** → ユーザーから直接背景情報を聞き取り、背景セクションを記載する
9. プルリクエストの説明欄を作成する。
   説明欄は手順 6 で判定した規模に応じて、以下のテンプレートを使い分ける。
    - 小規模の場合: シンプルテンプレートを使用
    - 中規模・大規模の場合: 詳細テンプレートを使用

    説明欄は、手順 2 で徹底的に確認した差分をもとに、テンプレートの TODO を埋める形で記述する。
    各セクションの見出しには絵文字をつけて見やすくする。
    PR タイトルは、差分内容から PR を一言でまとめたものを生成する。
    説明欄の内容は一時ファイル（例: `/tmp/pr_body.md`）に保存する。
    PR がない場合は `gh pr create --draft --base [ベースブランチ] --title "[PRタイトル]" --body-file [一時ファイルパス]` で作成する。
    既存 PR がある場合は `gh pr edit [PR番号] --body-file [一時ファイルパス]` で更新する。
    現在のブランチに対して既にプルリクエスト（手順 7 で確認）が存在する場合は更新し、存在しない場合はドラフトとして作成する。
    一時ファイルは作業完了後に削除すること。

### シンプルテンプレート（小規模用）

```md
<!-- I want to review in Japanese. -->

## 🚀 概要

TODO: プルリクエストの概要をここに記載する

## 💭 背景

TODO: なぜこの変更が必要になったのか、解決したい課題を簡潔に記載する。Notionリンクがある場合はリンクを併記し、Notionの内容を要約して記載する。

## ✨ できるようになったこと

TODO: このプルリクエストによって実現した価値や解消した課題を箇条書きで記載する。

## 👀 変更内容

TODO: このプルリクエストの変更内容を構造化された形で記載する
```

### 詳細テンプレート（中規模・大規模用）

```md
<!-- I want to review in Japanese. -->

## 🚀 概要

TODO: プルリクエストの概要をここに記載する

## 💭 背景

TODO: なぜこの変更が必要になったのか、解決したい課題を簡潔に記載する。Notionリンクがある場合はリンクを併記し、Notionの内容を要約して記載する。

## ✨ できるようになったこと

TODO: このプルリクエストによって実現した価値や解消した課題を箇条書きで記載する

## 👀 変更内容

### 📊 データ層

TODO: テーブル定義の変更をテーブル形式で記載する。ER図は関係性が複雑な場合のみ追加する。該当しない場合はセクションごと省略する。

### 🔌 API/インターフェース

TODO: エンドポイントや外部インターフェースの追加・変更を記載する。該当しない場合はセクションごと省略する。

### 🏗️ 設計変更

TODO: アーキテクチャやモジュール構成に大きな変更がある場合、図と説明を記載する。AsIs/ToBeは大きな構造変更がある場合のみ使用する。該当しない場合はセクションごと省略する。

### 🔮 処理フロー

TODO: 複雑なビジネスロジックの追加・変更がある場合、フローチャートと説明を記載する。該当しない場合はセクションごと省略する。

### 🧪 テスト

TODO: 追加・変更したテストの観点を簡潔に記載する。該当しない場合はセクションごと省略する。

## ⚠️ 注意事項

TODO: 破壊的変更、マイグレーション手順、デプロイ時の注意点などを記載する。該当しない場合はセクションごと省略する。
```

## ✅ 完了条件

- PR がドラフトとして作成（または既存 PR が更新）されている
- 一時ファイルが削除されている

## 📦 アウトプット

- 作成（または更新）された PR の URL

## ⚠️ 注意事項

- 「✨ できるようになったこと」のセクションは大げさに書かずに、シンプルに記載する
- 内容は構造化して理解しやすいようにまとめる
- 各セクションには文の先頭に絵文字をつけて見やすくする
- 内容が空になるセクションはセクションごと記載を省略する
- AsIs/ToBe は大きな構造変更がある場合のみ使用する
