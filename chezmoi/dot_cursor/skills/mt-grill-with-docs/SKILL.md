# 文書付き徹底ヒアリング

`mt-grill-me` の徹底質問に、用語集と ADR の更新を加える。認識合わせと文書化を同時に進める。

## 🧠 前提知識

- ヒアリング本体: [../mt-grill-me/SKILL.md](../mt-grill-me/SKILL.md)
- 用語集形式: [CONTEXT-FORMAT.md](CONTEXT-FORMAT.md)
- ADR 形式: [ADR-FORMAT.md](ADR-FORMAT.md)
- 用語集は実装詳細を書かない。ADR は取り消せない・驚く・トレードオフがある判断だけを残す

## 🏃 ステップ

### 1. 対象と文書の所在を確認する

- ヒアリング対象（計画・設計・仕様の論点）を確認する
- 既存の `CONTEXT.md` / `CONTEXT-MAP.md` / `docs/adr/` を探す
- なければ、最初の用語または ADR が確定した時点で遅延作成する

### 2. 徹底ヒアリングを行う

- [../mt-grill-me/SKILL.md](../mt-grill-me/SKILL.md) に従う
- 質問は 1 つずつ、番号付き 3 選択肢と推奨度を添える
- コードベース探索で答えられることは質問せず調査する
- 確認が済むまで実装・作成・状態遷移を進めない

### 3. 用語をその場で確定する

- 曖昧語・衝突語を見つけたら、その場で正規用語を提案する
- 確定したら [CONTEXT-FORMAT.md](CONTEXT-FORMAT.md) に従い `CONTEXT.md` を更新する
- 実装詳細・仕様・設計案を用語集に混ぜない

### 4. 判断を ADR に残す

次の 3 条件をすべて満たす判断だけ ADR を提案する:

1. 後から覆すコストが高い
2. 文脈なしだと将来の読み手が驚く
3. 実在するトレードオフから選んだ

- 提案し、ユーザーが望む場合のみ [ADR-FORMAT.md](ADR-FORMAT.md) で作成する
- 一時的な都合や自明な選択は ADR にしない

### 5. 結果を要約する

- 確定した用語
- 作成・更新した ADR
- 未解決の論点
- 次に進める範囲

## ✅ 完了条件

- 主要論点について共通認識に達している、または未解決点が明示されている
- 確定した用語が `CONTEXT.md` に反映されている
- 条件を満たす判断が ADR として提案または記録されている

## 📦 アウトプット

- 更新された `CONTEXT.md`（必要時）
- `docs/adr/NNNN-slug.md`（必要時）
- ヒアリング結果の要約

## ⚠️ 注意事項

- 原文の機械翻訳や丸写しをしない
- 用語集を仕様書や実装メモの置き場にしない
- ADR を量産しない
